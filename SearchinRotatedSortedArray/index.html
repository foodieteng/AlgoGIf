Why does Binary Search still work?

A rotated sorted array like [4,5,6,7,0,1,2] looks broken, but it has a powerful invariant:
At every step, at least one half — [lo..mid] or [mid..hi] — is perfectly sorted.

We can identify the sorted half by checking nums[lo] <= nums[mid]. Once we know which half is sorted, we check if the target falls in that sorted range. If yes, we narrow the search there. If not, we go to the other half.

Left Half Sorted

When nums[lo] <= nums[mid], the subarray [lo..mid] is in ascending order. If target ∈ [nums[lo], nums[mid]], set hi = mid. Otherwise lo = mid + 1.

Right Half Sorted

When nums[lo] > nums[mid], the rotation pivot is in the left half, so [mid..hi] is sorted. If target ∈ (nums[mid], nums[hi]], set lo = mid + 1. Otherwise hi = mid.

Why hi = mid, not mid − 1?

Using the lo < hi loop pattern, mid always satisfies lo ≤ mid < hi. Setting hi = mid keeps mid as a candidate and always shrinks the window. The loop converges to one element, checked after exit.

Complexity

Time: O(log n) — each iteration halves the search window.
Space: O(1) — only three integer variables.
